枚举内存布局：

枚举的内存占用取决于其变体的最大内存需求。枚举的内存布局需要能够容纳所有变体中的最大数据量，同时还需要存储一个判别式（discriminant）来区分不同的变体。

内存布局

  1. 判别式：

    • 每个枚举变体都有一个判别式，用于区分不同的变体。
    • 判别式通常是一个整数，其大小取决于枚举变体的数量。对于大多数情况，判别式占用 1 个字节（u8）或 2 个字节（u16），但如果有更多的变体，可能会占用更大的空间。

  2. 变体数据：

    • 每个变体的数据需要存储在枚举的内存中。
    • 枚举的总内存大小是所有变体中最大数据大小加上判别式的大小。

通过使用 std::mem::size_of_val 函数，你可以查看具体实例的内存占用情况，从而更好地理解枚举的内存布局。

&str vs String:

&str (字符串切片)

  1. 不可变性：&str 是一个不可变的字符串切片，通常用于表示一段只读的字符串数据。
  2. 借用特性：&str 是一个引用，它借用了某个地方的数据，不拥有数据的所有权。这意味着它的生命周期受到所引用数据的限制。
  3. 栈上分配：由于&str是引用类型，它自身占用的内存很小，通常只包含指向实际字符串数据的指针和长度信息，因此可以在栈上高效地传递。
  4. 性能优势：因为不需要复制字符串数据，使用&str可以提高性能，尤其是在频繁传递或处理大量文本数据时。
  5. 应用场景：适合用于不需要修改的字符串数据，如函数参数、配置项等。

String

  1. 可变性：String 类型是可变的，允许在其生命周期内添加、删除或修改内容。
  2. 所有权：String 拥有其数据的所有权，这意味着当String实例超出作用域时，其数据将被自动释放。
  3. 堆上分配：String 实际的数据存储在堆上，这使得它可以动态增长，但也意味着创建和销毁String实例可能涉及更多的内存管理开销。
  4. 灵活性：由于String是可变的，并且可以动态调整大小，因此它提供了更多的灵活性，适用于需要构建或修改字符串的场景。
  5. 应用场景：适合用于需要修改或构建的字符串数据，如从用户输入中读取文本、拼接字符串等。

插入(Insert)：

  4. 由于是修改原来的字符串，所以字符串必须可变，要有mut

注意：

  1. 插入的时候不能越界
  2. 插入的位置必须是合法的边界

Rust lesson 7 - 引用

发表于 2024-10-21 | 分类于 技术/Rust 开发

不可变引用（Immutable Reference）：

  1. 通过不可变引用，可读但不能改数据。
  2. 一个变量可以有多个不可变引用，但不可变引用不能与可变引用共存。

可变引用：

  1. 通过可变引用，可读并修改数据。
  2. 一个变量某一时刻只能有一个可变引用，且不能与不可变引用共存

引用规则：

  1. 同一时间内，一个变量只能有一个可变引用 或 多个不可变引用。
  2. 引用必须是有效的。

悬垂指针（dangling pointers):

是指一个指针或引用指向的内存已经被释放或不再有效的情况，是个严重错误。

代码示例：

生命周期（life time)：

“生命周期”是指引用（包括不可变引用和可变引用）的有效期,即从引用被创建开始，直到引用不再被使用为止的时间段。帮助编译器确保引用总是有效的，并且不会导致悬垂指针（dangling pointers）或数据竞争等问题。

生命周期注解：

可以通过生命周期注解（lifetime annotations）来显式地指定引用的有效时间范围。生命周期注解通常用 'a 形式的符号表示，其中 'a 是一个生命周期参数。

为什么要关心生命周期？

  1. 防止悬垂指针：确保引用在其生命周期内始终有效，不会指向已被释放的内存。
  2. 避免数据竞争：确保在多线程或多引用情况下，数据不会被同时修改而导致竞争条件。
  3. 确保内存安全：通过严格的生命周期管理，Rust 编译器可以静态地验证代码的内存安全性。

生命周期规则：

  1. 每个引用都有一个生命周期：引用的生命周期至少覆盖了它的作用域。
  2. 引用的生命周期不能超过被引用的数据的生命周期：如果一个引用指向的数据被释放，则该引用无效。
  3. 可变引用必须是唯一的：在一个作用域内，只能有一个可变引用，不能同时存在多个可变引用。

如何指定生命周期：

  1. 隐式生命周期：如果编译器能够推断出生命周期，可以省略生命周期注解。
  2. 显式生命周期注解：在函数签名中显式指定生命周期参数，如 fn longest<'a>(x: &'a str, y: &'a str) -> &'a str。

一般来讲，Rust 编译器通常能够自动推断出生命周期，但在某些情况下，需要显式地指定生命周期注解来帮助编译器理解代码意图。

Rust lesson 6 - 所有权

发表于 2024-10-20 | 分类于 技术/Rust 开发

所有权零开销内存回收的一种高效实现方式

Rust是一种系统变成语言，其设计目的是确保内存安全并防止数据竞争，而不是依赖垃圾回收器。这种内存安全性主要通过所有权系统来实现

内存回收

为什么要内存回收？

因为硬件内存大小受限（就只有这么多)，数据泄露，隐私安全（回收以避免其他软件读取数据）

主流编程语言内存回收机制对比：

静态语言：

  1. 在编译时对变量进行检查和确定的语言。
  2. 例如：c 、c++ 、 rust

动态语言：

  1. 在运行时进行类型检查和确定的语言。
  2. 例如：js 、python

差异对比：

各种编程语言都有属于自己内存回收机制，有些是手动的，有些是自动的，这里只介绍Rust 的，其它语言暂不赘述，请自行查找学习。

Rust 内存回收：

内存管理方式：所有权系统（Ownership System)

特点：

  1. Rust使用所有权系统进行内存管理，编译期在编译时通过静态分析来确保内存安全。
  2. 每个值都有一个所有者，在任何时候都只能有一个有效所有者
  3. 通过借用（引用）机制来共享数据，同时保证数据竞争和悬垂指针安全。

优点：

  1. 在编译时保证内存安全，没有运行时开销。
  2. 避免了数据竞争和悬垂指针。

缺点：

  1. 需要使用人员理解和遵循所有权和借用规则，学习曲线较陡。

所有权规则：

Rust为了确保内存安全并防止数据竞争，使用所有权系统来实现，而不依赖垃圾回收器。

规则：

  1. 每一个值都有一个所有者(owner)。
  2. 在任一时刻，值只能有一个所有者。
  3. 当所有者离开作用域（scope)，值会被丢弃（drop)。

  3. 都必须明确指定类型
  4. 两者都要求必须使用常量表达式进行赋值，即必须是编译期能计算出的值
  5. 两者都不支持重定义（遮蔽），这点和变量不同

3.4 静态变量与常量的区别：

  1. 常量在编译时被内联，但静态变量不会。在整个程序中静态变量只有一个实例，也就是说所有引用都指向同一个地址。
    const A : i32 = 1；就是地址A（0x456）的空间里面装了 1 这个值.
    static A : i32 = 1; 就是地址A（0x789）的空间里装了0x123这个地址，指向了0x123这个地址，0x123里面刚好装了 1 这个值，然后读A就读到0x123里面那个1，也就是指针指向值所在的内存地址。
  2. 常量不可变，而静态变量和普通变量一样，默认不可变，但可以通过 mut 关键字定义为可变。

4.作用域和遮蔽（variable shadowing)

变量绑定有一个作用域(scope)，它被限定只在一个代码块(block)中生存（live)。代码块是一个被 “{ }”包围的语句集合。另外也允许变量遮蔽（variable shadowing)。
